Данный домумент описывает конечные точки (endpoints), которые должен поддерживать ЛИС со своей стороны для интеграции с ПАКС OneCell.

OneCell будет делать запросы на данные конечные точки для передачи информации из ПАКС в ЛИС.

Получение набора случаев из LIS по идентификаторам

Получение набора случаев из LIS для их обновления на стороне PACS, либо для их выгрузки в PACS по его инициативе.

При поиске всегда передаётся массив, где каждый элемент один или несколько идентификаторов для фильтрации.

При передаче нескольких условий поиска в рамках одного объекта, сервер должен искать объект, который удовлетворяет всем условиям поиска (AND):

[
  // Ищем случай, где id_случая=B001 И номером_заказа=345
  { "caseId": "B001", "orderNumber": "345" },

  // Также ищем случай, где id_случая=B002
  { "caseId": "B002" }
]

Импорт списка импортированных ММК в LIS

ЛИС должен принимать от OneCell набор ММК (мединиского медиа контента - микро, макро слайдов и документов) и сохранять в своей базе данных для последующего вывода врачу прямых ссылок на слайды и документы в интерфейсе OneCell.

Операция выполнена успешно?

boolean

🆕Добавлено в версии API 1.10

Должна ли система, отправившая запрос, повторить его выгрузку ещё раз в случае ошибки.

Если success = true, то значение всегда retryable = false. Если success = false, то:

  1. retryable = true, если принимающая система ДОЛЖНА попробовать выгрузить данный объект повторно через время (например, если на принимающей стороне произошла ошибка обращения к внешнему серверу или к БД)
  2. retryable = false, если принимающая система НЕ ДОЛЖНА выполнять повторные попытки выгрузки данной сущности (напр. поля в сущности указаны некорректно и повторные попытки выгрузки такого объекта никогда не приведут к успешному результату)

🆕 Выгрузка массива Запросов на Дозаказ (ЗнД)

По данному эндпоинту ПАКС OneCell будет выгружать в ЛИС массив Запросов на Дозаказ.

ЛИС обязан ответить на каждый запрос в массиве индивидуально одним из следующих образов:

  1. Если ЗнД сохранена успешно, то ЛИС должен ответить следующим образом, а также создать и выгрузить отдельным запросом ВСЕ стёкла, указанные в ЗнД:
      {
    "success": true,
    "retryable": false
  }
  2. Если ЗнД не сохранена из-за временной ошибки (напр. внешний сервер недоступен), то ЛИС должен ответить следующим образом:
      {
    "success": false,
    "retryable": true, // <-- TRUE говорит о том, что OneCell ДОЛЖЕН повторить попытку выгрузки этого объекта через 5 минут
    error: { ... здесь информация об ошибке (как в примере ниже) ... } 
  }
  3. Если ЗнД не сохранена из-за некорректного запроса со стороны OneCell (напр. указанное стекло отсутствует в ЛИС), то ЛИС должен ответить следующим образом: 
      {
    "success": false,
    "retryable": false, // <-- FALSE говорит о том, что OneCell НЕ ДОЛЖЕН делать повторных попыток выгрузки этого объекта, а вместо этого перевести ЗнД в статус "Ошибка"
    error: { ... здесь информация об ошибке (как в примере ниже) ... } 
  }

Если success = true, то значение всегда retryable = false. Если success = false, то:

  1. retryable = true, если принимающая система ДОЛЖНА попробовать выгрузить данный объект повторно через время (например, если на принимающей стороне произошла ошибка обращения к внешнему серверу или к БД)
  2. retryable = false, если принимающая система НЕ ДОЛЖНА выполнять повторные попытки выгрузки данной сущности (напр. поля в сущности указаны некорректно и повторные попытки выгрузки такого объекта никогда не приведут к успешному результату)
